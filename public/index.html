Harbor Framework

Harbor is a Ruby web framework founded on the principles of composition, testability, and consistency. It leverages the power of existing technologies like Rack and MailFactory.

While Harbor will eventually be compatible with JRuby only (we'll wait until the O/RM is ready), it also emphasizes using Ruby (the language) where practical as opposed to a bunch of custom helpers. It generally makes things a little more verbose, but it's a lot more explicit, and easier for new-comers to grasp if they have a good working understanding of Ruby.

Speaking of being easy on new-comers, Harbor works practically out of the box; it's only dependencies being Rack and MailFactory. Once you have JRuby setup, you can just gem install warbler to get a JRuby version of Rack along with the rackup command. Startup for JRuby is quite a bit slower (a few seconds), but nothing too painful. For now, if you just want to experiment a bit with Harbor under Ruby 1.8.x (tested on 1.8.6), you can just use the thin web-server (gem install thin).

We're also steadily working an ORM (Worm, internally). The goal is to tap into all that Java performance behind the scenes, while keeping your work all Ruby all the time. Worm is easy to get started with too. You don't even need RSpec (we're going back to our xUnit roots with Test::Unit). Just gem install jdbc-sqlite3 jdbc-mysql jdbc-hsqldb and you're good to go. It doesn't do everything, but everything is built in. Still, give us just a bit more time before you go trying to use it for anything useful. Harbor is stable and vetted on production applications. Worm, not so much yet.

If you'd like to get started with Harbor, or just want to play around with Worm, visit Github and checkout the work going on at WieckLabs. We're putting together better documentation, too, but you can take a peek at what we have so far here. 

    1 # Let's create an application for managing users, logging in, all that good stuff.
    2 # You know, the sort of thing there are a million Rails plugins for, and none of
    3 # them work the way you want to.
    4 class UserManagement < Harbor::Application
    5 
    6   # Harbor leaves your application layout almost entirely up to you.
    7   # Just the same, the default generators provide a convention that works for us
    8   # to help you get started.
    9   # One convention is the ::default_routes method defined on your Application class.
   10   # We'll assume that you pass in a Harbor::Container (a "Inversion of Control"
   11   # container) to give you access to common services like loggers, mailers, etc.
   12   def self.routes(services)
   13     
   14     # You don't have to use the Harbor::Router, but it's there if you want it.
   15     # A Harbor::Application only requires that you pass your router in to it's
   16     # initializer, and that it has a #match method that accepts a Rack env Hash
   17     # and returns a lambda that accepts a Harbor::Request and a Harbor::Response.
   18     Harbor::Router.new do
   19       
   20       # get, post, put, delete. It's all there buddy! A little more typing, but
   21       # aren't explicit routes so nice? Here we're being lazy and skipping using
   22       # any controller at all, just rendering our View directly.
   23       # Harbor only has two fundamental objects. The Request and the Response.
   24       # Your params and Session is available through the Request (can't have a
   25       # Session without one right?), and you #puts, #send_file, set your #cookies
   26       # or #render on your Response.
   27       get("/login") { |request, response| response.render "/account/login" }
   28       
   29       # The Router#using method takes a Container and a class to instantiate. Really
   30       # it's just a helper method that calls
   31       # Harbor::Container#register(controller.name, controller)
   32       # and wraps your lambda so that the request, response, and whatever other
   33       # accessors you might declare on your controller are set for you automatically.
   34       using services, UserManagement::Account do
   35         post("/login") do |account, params|
   36           account.login(params[:user][:email], params[:user][:password])
   37         end
   38         
   39         get("/account") { |account| account.index }
   40       end
   41 
   42       # Response#redirect sets your response-headers to redirect the user to the
   43       # new location. If you want to jump out of the request right away call
   44       # #redirect! and your current method will #throw an :abort_request symbol
   45       # the Harbor::Application will catch.
   46       get("/") { |request, response| response.redirect!("/account") }
   47     end
   48   end
   49   
   50   # Just a little more convention since Harbor::Application normally requires
   51   # something that quacks like a Router.
   52   def initialize(services = Harbor::Container.new)
   53     # And here we satisfy the base-class's initializer by providing the Router
   54     # returned by ::default_routes.
   55     super(self.class.routes(services))
   56   end
   57 end

   59 # Ok, we have our Application ready to go, but we'll need to setup our controller.
   60 # Notice Harbor doesn't care what your controller is. There's no Harbor::Controller
   61 # class. The most we do is provide the Harbor::Container to make it easy to write
   62 # your own. How do we get from A to B? Here's some pseudo-code that might help.
   63 # Here's basically how the Harbor::Application serves your request (env is a Rack
   64 # env Hash):
   65 #
   66 #   handler = router.match(env)
   67 #   request = Harbor::Request.new(env)
   68 #   response = Harbor::Response.new(request)
   69 #   handler.call(request, response)
   70 #   return response.to_s
   71 #
   72 # So basically, you mutate the Response. Or not. Don't do anything with it if you
   73 # want. That would be a kind of pointless application, but it's your call.
   74 class UserManagement::Account
   75   
   76   # What is this sir? Well, remember how I said the Harbor::Container helps you out?
   77   # Wait, let's take a step back. What is an "Inversion of Control Container"?
   78   # Simple. It's a form of "Dependency Injection" (DI). Ok, what's DI?
   79   # It's an Object Oriented Programming Pattern that I'm not going to be able to
   80   # do justice here. Instead, let me show you what it does, and you can read up on
   81   # the details later.
   82   #
   83   #   container = Harbor::Container.new
   84   #   
   85   #   class Foo
   86   #     attr_accessor :bar
   87   #   end
   88   #
   89   #   class Bar
   90   #     attr_writer :baz
   91   #   end
   92   #
   93   #   class Zoo
   94   #   end
   95   #
   96   #   container.register("foo", Foo)
   97   #   container.register("bar", Bar)
   98   #   container.register("baz", Zoo)
   99   #   
  100   #   example = container.get("foo")
  101   #    
  102   #   assert_equal("Zoo", example.bar.baz.class.name)
  103   #
  104   # Notice the "baz"/Zoo descrepency. It doesn't matter what a registration is,
  105   # only what you decide to call it. So basically Harbor::Container#get uses the
  106   # attr_writers defined on your classes to match up dependencies with
  107   # registrations. The major limitation here is that your constructors have to be
  108   # parameterless since Ruby doesn't declare types so we wouldn't know what to
  109   # give them without something like ParseTree.
  110   #
  111   # Anyways, back to the story. Harbor::Router#using, in conjunction with the
  112   # Harbor::Container can setup your controllers for you. Just declare writers
  113   # for :request and :response. Of course readers are convenient to, so we just
  114   # used attr_accessor.
  115   #
  116   # This way say you needed to provide your controller a logger, you could just
  117   # do something like this:
  118   #
  119   #   module LogHelper
  120   #     def debug(message)
  121   #       logger.debug(message) if logger
  122   #     end
  123   #   end
  124   #
  125   #   class Account
  126   #     attr_accessor :logger
  127   #     include LogHelper
  128   #     def index
  129   #       debug("You've found the /index action!")
  130   #     end
  131   #   end
  132   #
  133   #   container.register("logger", Logger.new)
  134   #
  135   # This makes our classes easier to test (you could stub out the logger, or in
  136   # this example, just not provide one), and it's an awfully convenient way to
  137   # register and consume services.
  138   attr_accessor :request, :response
  139   
  140   def login(email, password)
  141     # You look like a trust-worthy sort!
  142     # Let's give you a sneak-peak at our ORM while we're
  143     # at it...
  144     me = orm.first(User) do |user|
  145       user.email.eq(email).and(user.password.eq(password))
  146     end
  147     
  148     if me
  149       request.session[:user_id] = me.id
  150       # Notice nothing else happens in this action, so we don't really care
  151       # about using the Response#redirect! variation.
  152       response.redirect("/account")
  153     else
  154       response.render("/account/login", :message => "Nein!")
  155     end
  156     
  157     # Remember, all that matters is what you do with Request and Response.
  158     # We don't care what you return from your handlers. We're not nosy like that.
  159     :badgers
  160   end
  161   
  162   def index
  163     me = begin
  164       orm.get(User, request.session[:user_id])
  165     rescue Harbor::Orm::ObjectNotFoundError
  166       # Ah, now we do want to escape this madhouse if you aren't who you say you are...
  167       response.redirect!("/login")
  168     end
  169     
  170     # Did I mention that you have pass your variable declarations to your views
  171     # explicitly? Keeps things easy to understand for the new guys. You do plan
  172     # to get rich and be able to stop maintaining this application someday right?
  173     response.render "/account/index", :me => me
  174   end
  175 end

  185 # Notice, Harbor::View::path is actually an Array of search paths. This way you
  186 # could package your application up as a gem and use all the views from the gem.
  187 # But you need a new awesome login page. Easy. Unshift another path, add an
  188 # "account" folder with a "login.html.erb" in it, and since it appears first on
  189 # the search path whenever we try to render "/account/login" we'll use your
  190 # sweet new view.
  191 #
  192 # This means Harbor is powerfully awesome. Or awesomely powerful. One of those.
  193 # Maybe even both. Basically you can overwrite individual views, just apply new
  194 # layouts, overwrite individual routes (by using a Rack::Cascade and simply
  195 # making sure you handle the route you want to overwrite before it gets to the
  196 # application that would normally handle it), or entire controllers.
  197 # That's what we like to call a "Port". Really it's just convention. Any Harbor
  198 # application can work with another without modification. Now just sit back and
  199 # imagine the possibilities...
